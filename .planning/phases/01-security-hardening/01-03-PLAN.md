Output: Two new columns in users schema (emailVerified, emailVerificationToken), updated register and login routes, a new GET /auth/verify/:token route, and a sendVerificationEmail() helper.

```typescript
export async function sendVerificationEmail(
  to: string,
  name: string,
  token: string,
): Promise<void> {
  const baseUrl = process.env.APP_BASE_URL ?? "http://localhost:5173";
  const verifyUrl = `${baseUrl}/verify-email?token=${encodeURIComponent(token)}`;
  await sendEmail({
    to,
    subject: "Verify your HypeSpace email address",
    html: `
      <p>Hi ${escapeHtml(name)},</p>
      <p>Click the link below to verify your email and activate your account:</p>
      <p><a href="${verifyUrl}">${verifyUrl}</a></p>
      <p>This link expires in 24 hours. If you didn't register for HypeSpace, you can ignore this email.</p>
    `,
  });
}
```

Use the existing `escapeHtml` or HTML-escape the name inline (per the XSS fixes already applied in the prior session). Import `escapeHtml` if it exists in the codebase, otherwise inline:
```typescript
function escapeHtml(str: string): string {
  return str.replace(/&/g, "&amp;").replace(/</g, "&lt;").replace(/>/g, "&gt;").replace(/"/g, "&quot;");
}
```

**Part D — Add GET /auth/verify/:token route in auth.ts:**

```typescript
// GET /auth/verify/:token — verifies email and redirects to login
router.get("/auth/verify/:token", async (req, res): Promise<void> => {
  const { token } = req.params;
  if (!token || typeof token !== "string") {
    res.status(400).json({ error: "Invalid verification token" });
    return;
  }

  const [user] = await db.select().from(usersTable)
    .where(eq(usersTable.emailVerificationToken, token));

  if (!user) {
    res.status(400).json({ error: "Invalid or expired verification token" });
    return;
  }

  await db.update(usersTable)
    .set({ emailVerified: true, emailVerificationToken: null })
    .where(eq(usersTable.id, user.id));

  // Redirect to login page with a success query param
  const baseUrl = process.env.APP_BASE_URL ?? "http://localhost:5173";
  res.redirect(`${baseUrl}/login?verified=true`);
});
```

**Part E — Add POST /auth/resend-verification route (optional but needed for good UX):**

```typescript
router.post("/auth/resend-verification", authLimiter, async (req, res): Promise<void> => {
  const { email } = req.body ?? {};
  if (!email || !isValidEmail(email)) {
    res.status(400).json({ error: "INVALID_INPUT" });
    return;
  }

  // Return same response whether user exists or not (prevents enumeration)
  const [user] = await db.select().from(usersTable).where(eq(usersTable.email, email.toLowerCase()));

  if (user && !user.emailVerified) {
    const token = crypto.randomBytes(32).toString("hex");
    await db.update(usersTable)
      .set({ emailVerificationToken: token })
      .where(eq(usersTable.id, user.id));
    try {
      await sendVerificationEmail(user.email, user.name, token);
    } catch (err) {
      req.log.error({ err }, "Failed to resend verification email");
    }
  }

  res.json({ message: "If that email is registered and unverified, a new verification link has been sent." });
});
```

Note: The `authLimiter` is already defined in `routes/index.ts` and applied to `/auth/*`. The resend endpoint is under `/auth/` so it is automatically rate-limited.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public internet → POST /auth/register | New user data crosses here; account squatting risk |
| Email client → GET /auth/verify/:token | One-time token in email link; account activation |
| Browser → POST /auth/login | Login attempt; unverified accounts must be blocked |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | Registration without verified identity | mitigate | Login blocked until email verification link clicked (per SEC-02) |
| T-03-02 | Elevation of Privilege | Account squatting (register with someone else's email) | mitigate | Email must be accessible to verify — owner controls delivery |
| T-03-03 | Information Disclosure | /auth/resend-verification reveals if email is registered | mitigate | Always return same response regardless of whether user exists or is verified |
| T-03-04 | Tampering | Verification token brute force | mitigate | 32-byte (256-bit) crypto.randomBytes token; rate-limited by authLimiter (20 req/15min); 64-char hex makes brute force infeasible |
| T-03-05 | Repudiation | Verification token reuse after successful verify | mitigate | Token set to NULL after verification; subsequent use returns 400 |
| T-03-06 | Denial of Service | Sending verification email fails, blocks registration | mitigate | Email send wrapped in try/catch; registration succeeds even if email fails; user can resend |
</threat_model>
